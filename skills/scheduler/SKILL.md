# scheduler

Создаёт автономные turns по cron-расписанию. Владелец НЕ должен быть
онлайн — триггер идёт сам, результат попадает в Telegram напрямую.

## Когда использовать

- "напомни мне каждый день в 09:00 проверить календарь"
- "раз в неделю сделай саммари vault"
- "по утрам в будни — план дня"

## Когда НЕ использовать

- Одноразовое напоминание → сохрани в vault (`memory write`), обычный
  user-turn завтра его подхватит по контексту.
- "через 5 минут" / "через час" → cron работает по wall-clock, не по
  delay; для одноразовых задержек используй `sleep` в клиентском коде
  (тут недоступен), а не cron.

## CLI

Все команды возвращают одну строку JSON на stdout. Ошибки — JSON на
stderr с `exit code`:

- `0` — ok
- `2` — usage
- `3` — валидация / cap-reached (≥ 64 schedules active)
- `4` — I/O (DB не создана)
- `7` — не найдено (ID не существует)

### Добавить schedule

```bash
python tools/schedule/main.py add --cron "0 9 * * *" --prompt "утро, что в календаре?"
python tools/schedule/main.py add --cron "0 9 * * 1-5" --prompt "план на день" --tz "Europe/Berlin"
python tools/schedule/main.py add --cron "0 10 * * 0" --prompt "еженедельный review vault"
```

### Посмотреть список

```bash
python tools/schedule/main.py list
python tools/schedule/main.py list --enabled-only
```

### Soft-delete / включить обратно

```bash
python tools/schedule/main.py rm 1        # enabled=0
python tools/schedule/main.py enable 1    # enabled=1
python tools/schedule/main.py disable 1   # enabled=0
```

### Историю (audit log)

```bash
python tools/schedule/main.py history --schedule-id 1 --limit 10
```

## Cron primer (5-field POSIX)

```
m  h  dom mon dow
│  │  │   │   │
│  │  │   │   └── день недели: 0=Вск .. 6=Суб (Mon=1, Sun=0 или 7? — НЕТ, 0..6 only)
│  │  │   └────── месяц: 1..12
│  │  └────────── день месяца: 1..31
│  └───────────── час: 0..23
└──────────────── минута: 0..59
```

Синтаксис:

- `*` — все значения
- `N` — одно значение
- `A-B` — диапазон (inclusive)
- `A,B,C` — список
- `*/N` — каждое N-е (шаг)
- `A-B/N` — шаг внутри диапазона

### Примеры

| Cron            | Смысл                                      |
|-----------------|--------------------------------------------|
| `0 9 * * *`     | каждый день в 09:00                        |
| `0 9 * * 1-5`   | будни (Пн-Пт) 09:00                        |
| `*/15 * * * *`  | каждые 15 минут                            |
| `0 9 1 * *`     | 1-го числа каждого месяца в 09:00          |
| `0 0 * * 0`     | воскресенье полночь                        |
| `0 10 * * 6`    | суббота 10:00                              |
| `0 */4 * * *`   | каждые 4 часа на 00 мин (00, 04, 08, 12, 16, 20) |

## Timezone

По умолчанию `UTC`. Переопределяется через `--tz "<IANA-name>"`:

- `UTC`, `Europe/Berlin`, `Europe/Moscow`, `America/New_York`, `Asia/Tokyo`
- Фиксированные смещения: `Etc/GMT+3` (обратите внимание: POSIX-инверсия
  знака — `Etc/GMT+3` это `UTC-3`, не `UTC+3`)

## Границы и поведение

- **Prompt — snapshot на момент `add`.** Редактирование vault не
  изменит prompt уже созданного schedule; нужно `rm` + `add`.
- **DST spring-forward:** если локальная минута не существует
  (например, `30 2 * * *` на переход EU→летнее 02:00→03:00), срабатывание
  этого дня пропускается. Warn-лог в daemon.
- **DST fall-back:** если локальная минута встречается дважды (переход
  с лета на зиму), срабатывание ровно один раз — на первой встрече (летнее смещение).
- **Catch-up:** если daemon был suspend/down > 1 часа (по умолчанию
  `SCHEDULER_CATCHUP_WINDOW_S=3600`), пропущенные срабатывания НЕ
  бэкфилятся. При старте daemon покажет recap: "пропущено N напоминаний".
- **Cap:** максимум `SCHEDULER_MAX_SCHEDULES=64` активных schedules.
  При превышении `add` возвращает exit 3 с `"error": "scheduler_schedule_cap_reached"`.
- **At-least-once доставка.** При перезапуске бота посреди отправки
  длинного ответа вы можете увидеть частично повторное сообщение. Это
  сознательная дисциплина — дубль напоминания мы предпочитаем потере.

## Что НЕ поддерживается

- Летер-нэйзы (`JAN`, `MON`) — только цифры.
- `@daily`, `@reboot`, `@yearly` и прочие сокращения — только 5-field.
- `DOW=7` для воскресенья — используй `0` (в некоторых cron-диалектах
  7=воскресенье принимается, мы ЯВНО отвергаем для единообразия).
